# Orchestrating TDD Skill

## Use when

- ユーザーが「TDDで実装して」「テスト駆動で」と言ったとき
- 新機能をTDDアプローチで実装するとき
- t_wadaのTDDプラクティスに従って実装を進めるとき

## 実行フロー

### Phase 1: 要件分析とテストケース分解

1. ユーザーの要件を確認
2. 要件をテスト可能な振る舞いに分解
3. テストケースの優先順位を決定（シンプルなものから）
4. TODOリストを作成

```markdown
## TDD TODOリスト

- [ ] ケース1: 最もシンプルな正常系
- [ ] ケース2: 次にシンプルな正常系
- [ ] ケース3: 境界値
- [ ] ケース4: エラーケース
- [ ] ケース5: エッジケース
```

### Phase 2: Red-Green-Refactorサイクル

TODOリストの各項目に対して以下を繰り返す:

#### RED（失敗するテストを書く）

1. テストケースを1つ書く
   - テスト名は日本語で振る舞いを記述
   - AAAパターン（Arrange-Act-Assert）で構造化
2. テストを実行して失敗を確認
3. エラーメッセージが想定通りか確認

```bash
pnpm test path/to/file.test.ts
```

#### GREEN（テストを通す最小限の実装）

1. テストを通す最小限のコードを書く
2. t_wadaの戦略を適用:
   - **仮実装**: まずハードコードで通す
   - **三角測量**: 2つ以上のテストで実装を導く
   - **明白な実装**: 自明な場合は直接実装
3. テストが通ることを確認

#### REFACTOR（リファクタリング）

1. Martin Fowlerの原則に従いリファクタリング
2. 重複の除去
3. 命名の改善
4. 構造の整理
5. テストが通ることを再確認

### Phase 3: 完了報告

```markdown
## TDD完了レポート

### 実装した振る舞い
- ケース1: [説明] ... DONE
- ケース2: [説明] ... DONE
- ...

### テスト結果
- テスト数: X
- 全テストPASS

### リファクタリング内容
- [実施した改善の概要]
```

## t_wadaの原則

### テストの粒度

- **1テスト1アサーション**を基本とする
- テストは仕様書として機能する粒度にする
- テスト名を読むだけで仕様が理解できること

### ステップの大きさ

- 不安なときは小さなステップで
- 自信があるときは大きなステップでもOK
- 迷ったらテストを増やす

### テストの順序

1. 最もシンプルな正常系から始める（退化テスト）
2. 正常系を徐々に複雑にしていく
3. 境界値テスト
4. 異常系・エラーケース

## フレームワーク対応

### Vitest

```typescript
import { describe, it, expect, vi } from 'vitest'

describe('機能名', () => {
  it('振る舞いの説明（日本語）', () => {
    // Arrange
    // Act
    // Assert
  })
})
```

### テスト実行コマンド

```bash
# EC
cd apps/ec && pnpm test path/to/file.test.ts

# Backend
cd apps/backend && pnpm test path/to/file.test.ts

# ウォッチモード（TDD向き）
cd apps/ec && pnpm test --watch path/to/file.test.ts
```

## 注意事項

- 各サイクルの開始前に現在のフェーズ（RED/GREEN/REFACTOR）を明示
- テスト実行結果を毎回表示
- TODOリストの進捗を更新
- リファクタリングフェーズではテストの追加は行わない
- 困ったらユーザーに相談（仕様の曖昧さ、設計判断）
