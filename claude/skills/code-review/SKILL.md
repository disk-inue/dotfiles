# Code Review Skill

## Use when

- PRのコードレビューを依頼されたとき
- 実装したコードの品質チェックを求められたとき
- リファクタリングの提案を求められたとき

## レビュー観点

### 1. コード品質

- **可読性**: 変数名、関数名は意図が明確か
- **シンプルさ**: 過度な抽象化や複雑なロジックがないか
- **DRY原則**: 不要な重複がないか（ただし過度な共通化は避ける）
- **SOLID原則**: 適切な責務分離ができているか

### 2. TypeScript/型安全性

- **any/unknown/never**: これらの型を避け、明示的な型を使用しているか
- **型推論**: 適切に型推論を活用しているか
- **ジェネリクス**: 必要に応じて使用しているか
- **nullチェック**: オプショナルチェーン、Nullish coalescing を適切に使用

### 3. React/Next.js固有

- **Server Component**: 可能な限りServer Componentで実装しているか
- **Client Component**: `use client`の使用は最小限か
- **コンポーネントサイズ**: 100行以内に収まっているか
- **Props設計**: 適切なProps定義、デストラクチャリング
- **hooks**: useEffect/useCallbackの依存配列は正しいか

### 4. GraphQL/Backend

- **N+1問題**: DataLoaderパターンを使用しているか
- **トランザクション**: 関連操作は$transaction内で実行しているか
- **バリデーション**: 入力値の検証は適切か
- **エラーハンドリング**: 適切なエラーオブジェクトを返しているか

### 5. パフォーマンス

- **バンドルサイズ**: 不要なimportがないか
- **メモ化**: 必要に応じてuseMemo/useCallbackを使用しているか
- **画像**: next/imageを使用しているか
- **遅延読み込み**: 適切にDynamic importを使用しているか

### 6. テスト

- **テストカバレッジ**: 重要なロジックにテストがあるか
- **テストの質**: 実装の詳細でなく振る舞いをテストしているか
- **エッジケース**: 境界値やエラーケースをカバーしているか

## 出力フォーマット

```markdown
## コードレビュー結果

### 必須対応
- [ファイル:行番号] 問題の説明
  - 修正提案

### 推奨対応
- [ファイル:行番号] 改善点
  - 改善案

### 良い点
- [具体的に良かった実装や設計]

### 質問・確認事項
- [意図が不明な箇所への質問]
```

## レビューの姿勢

- 批判ではなく改善提案を心がける
- コードの意図を理解しようとする
- 過度な指摘は避け、重要な点に集中する
- 良い実装は積極的に褒める
